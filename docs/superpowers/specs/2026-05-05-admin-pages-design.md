# Admin Pages Design

## 背景

当前项目已经完成 Vue 客户端、Hono 服务端、认证、刷新令牌、登录态恢复，以及受
Bearer token 保护的系统用户、部门、角色和资源 API。登录后的 `/` 当前仍是一个
“欢迎回来”工作台，没有承载后台管理能力。

系统模块后端已经提供：

- `/api/system/users` 用户分页列表和 CRUD。
- `/api/system/departments` 部门分页列表、树查询和 CRUD。
- `/api/system/roles` 角色分页列表和 CRUD。
- `/api/system/resources` 资源分页列表、树查询和 CRUD。

第一版后台管理页面需要把这些已有系统模块可视化出来，但暂不进入新增、编辑、删除等
完整 CRUD 表单。

## 目标

1. 将登录后的客户端体验升级为后台管理壳层。
2. 使用左侧导航 + 主工作区的信息架构。
3. 新增用户、部门、角色、资源四个系统管理页面。
4. 四个页面都接真实 API 数据，不使用静态占位数据。
5. 用户和角色使用服务端分页列表。
6. 部门和资源使用只读树状表格，不分页。
7. 支持各页面必要筛选、加载中、空态、错误态和手动刷新。
8. 保持第一版范围在“浏览和筛选”，不实现新增、编辑、删除和详情抽屉。

## 非目标

第一版不实现：

- 新增、编辑、删除、批量操作或导入导出。
- 用户部门和角色关联选择表单。
- 角色资源授权树编辑。
- 资源树拖拽排序、动态菜单生成或权限聚合。
- 部门树拖拽排序或组织架构编辑。
- 移动端专项后台体验。
- 审计日志、回收站、恢复删除或高级权限判断。

后台主体验面向桌面浏览器。小屏只做最低限度兜底，避免内容重叠或页面不可用；不为
移动端单独设计抽屉导航、手势交互或移动端优化表格。

## 路由与鉴权

继续使用 `vue-router/vite` 文件路由。

新增后台页面：

- `/system/users`
- `/system/departments`
- `/system/roles`
- `/system/resources`

`/` 作为登录后的默认入口，可以重定向到 `/system/users`，或渲染同一个用户管理页面。
实现时优先选择与文件路由最自然的方式，避免引入额外路由配置复杂度。

鉴权守卫从只保护 `/` 扩展为保护全部非认证页。认证页仍为：

- `/login`
- `/register`

未登录访问任意后台页面时，尝试通过 refresh cookie 恢复登录态。恢复失败则跳转
`/login`，并保留原目标地址到 `redirect` 查询参数。已登录访问登录或注册页时继续
跳转后台默认入口。

## 后台壳层

新增 `AdminLayout.vue`，负责后台通用外壳：

- 左侧固定导航。
- 品牌区显示 `Rev30` 和“后台管理”。
- 导航分组为“系统管理”。
- 导航项包含用户、部门、角色、资源，并根据当前路由高亮。
- 主工作区承载当前页面。
- 顶部工具区显示当前用户摘要、主题切换和退出登录。
- 退出登录继续复用现有 `logout` 请求、auth store 和登录页跳转行为。

壳层使用 Naive UI 与 Tailwind CSS v4。视觉风格保持后台工具的密度和克制：不做 hero，
不做营销式大卡片，不添加装饰性渐变或复杂动效。

## 用户管理页

路由：`/system/users`

数据来源：`GET /api/system/users`

查询能力：

- `keyword`
- `status`
- `page`
- `pageSize`

展示字段：

- 用户名
- 昵称
- 联系方式，优先显示邮箱，其次手机；都为空时显示 `-`
- 状态，使用 Naive UI `NTag`
- 部门摘要
- 角色摘要
- 创建时间

部门和角色使用紧凑标签或文本摘要展示。数量较多时可以截断为前几个名称加数量提示，
避免表格行过高。

筛选提交后回到第 1 页并重新请求。分页变化重新请求当前筛选条件下的数据。

## 部门管理页

路由：`/system/departments`

数据来源：`GET /api/system/departments/tree`

第一版使用只读树状表格，不分页。

筛选能力：

- `keyword`，前端匹配部门名称和部门编码。
- `status`，前端按启用或禁用过滤。

展示字段：

- 部门名称
- 部门编码
- 状态，使用 Naive UI `NTag`
- 排序
- 创建时间

树过滤规则：

- 命中节点直接展示。
- 子节点命中时保留祖先节点，保证层级上下文不丢失。
- 未命中的兄弟节点不展示。

第一版可以提供“全部展开 / 全部收起”轻量操作，不做拖拽排序、树编辑或懒加载。

## 角色管理页

路由：`/system/roles`

数据来源：`GET /api/system/roles`

查询能力：

- `keyword`
- `status`
- `page`
- `pageSize`

展示字段：

- 角色名称
- 角色编码
- 状态，使用 Naive UI `NTag`
- 用户数
- 排序
- 创建时间

角色列表不展开资源授权。资源授权属于后续角色编辑或详情能力。

筛选提交后回到第 1 页并重新请求。分页变化重新请求当前筛选条件下的数据。

## 资源管理页

路由：`/system/resources`

数据来源：`GET /api/system/resources/tree`

第一版使用只读树状表格，不分页。

筛选能力：

- `keyword`，前端匹配资源名称、编码、路径和外链地址。
- `type`，前端按资源类型过滤。
- `status`，前端按启用或禁用过滤。

展示字段：

- 资源名称
- 资源编码
- 类型
- 路径或外链地址
- 是否隐藏
- 状态，使用 Naive UI `NTag`
- 排序
- 创建时间

资源类型展示为中文标签：

- `directory`：目录
- `menu`：菜单
- `external`：外链
- `action`：操作

树过滤规则：

- 命中节点直接展示。
- 子节点命中时保留祖先节点，保证层级上下文不丢失。
- 未命中的兄弟节点不展示。

第一版可以提供“全部展开 / 全部收起”轻量操作，不做资源树编辑、拖拽排序、动态菜单生成
或角色授权。

## 数据流

继续复用现有 Hono RPC client：

```ts
import { api } from '../api'
```

请求通过现有 `authFetch` 自动携带 Bearer token。401、token refresh 和本地登录态清理
继续由 `authFetch` 和路由守卫负责，列表页不重复实现登出或刷新 token 逻辑。

用户和角色页面：

- 页面状态保存筛选条件、页码和每页数量。
- 查询参数变化后调用服务端列表接口。
- 服务端返回 `list`、`total`、`page` 和 `pageSize`。

部门和资源页面：

- 页面加载和刷新时拉取完整树。
- 筛选在前端纯函数中完成。
- 过滤函数不修改原始树节点。
- 过滤结果用于表格展示和结果数量统计。

使用 Pinia Colada 管理异步请求、加载状态、错误状态和刷新。页面内可以根据实际 API
调用形态选择 query key，避免把筛选逻辑隐藏进过度抽象的通用组件。

## 组件边界

第一版只抽必要公共组件。

新增：

- `AdminLayout.vue`

不新增：

- `AdminPageHeader.vue`
- `AdminTableToolbar.vue`
- 自定义 `StatusTag.vue`

原因：

- 四个页面的查询字段不同，通用筛选栏容易变成配置驱动组件。
- 页面标题、数量和刷新按钮逻辑很薄，暂不值得抽公共组件。
- Naive UI 已提供 `NTag`，状态展示直接使用即可。

每个页面自己组织标题行、筛选栏、表格、分页和刷新按钮。树过滤逻辑抽成纯函数，便于
单元测试和后续复用。

## 错误处理

页面只在边界展示必要状态：

- 加载中：使用 Naive UI 表格 loading 状态。
- 空数据：表格空态显示“暂无数据”或筛选后无结果。
- 请求失败：页面上方显示 `NAlert`。
- 错误文案优先使用接口返回的 `message`；没有稳定 message 时显示通用失败文案。
- 筛选重置恢复默认查询。
- 用户和角色筛选重置后回到第 1 页。

不添加宽泛 try/catch、复杂 fallback 或页面级 token 处理。

## 测试设计

按 TDD 执行，先写失败测试再写实现。

客户端测试重点：

- 路由守卫保护所有后台页面；未登录访问 `/system/users` 跳转登录并保留 redirect。
- 已登录访问认证页跳转后台默认入口。
- 后台壳层显示导航、当前用户、主题切换和退出登录。
- 退出登录清理本地 session 并跳转登录页。
- 用户页发起正确查询，渲染用户列表、状态、部门、角色和分页。
- 用户页筛选和分页变化会改变请求参数。
- 角色页发起正确查询，渲染角色列表、状态、用户数和分页。
- 角色页筛选和分页变化会改变请求参数。
- 部门树过滤命中子部门时保留父部门，状态筛选正确过滤。
- 资源树过滤支持关键词、类型、状态，并保留命中节点的父级上下文。

不为纯样式、Naive UI 内部行为或简单导入整理添加冗余测试。

实现完成后至少运行：

```bash
pnpm --filter @rev30/client test
pnpm --filter @rev30/client typecheck
pnpm check
```

如果实现过程中修改共享 schema 或服务端类型合同，再补充运行对应 workspace 的测试和
typecheck。
