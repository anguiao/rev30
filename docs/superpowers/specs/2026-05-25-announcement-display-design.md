# 通知公告展示功能设计

## 背景

Rev30 已完成后台“内容管理 - 通知公告”的基础管理能力。现有能力包括公告类型、草稿/发布/归档生命周期、置顶、Tiptap JSON 正文存储、`contentText` 派生搜索、管理端列表和表单，以及基于 `content:announcement:*` 的管理权限控制。

上一版设计明确不包含前台或业务侧公开读取接口。本轮目标是在保持管理能力边界清晰的前提下，补齐登录用户查看通知公告的展示能力，并补充公告的可见范围配置。

## 目标

- 支持发布人在管理端配置公告可见范围。
- 支持全员可见，以及指定用户、部门、角色可见。
- 部门使用树形选择，但只按精确选中的部门判断可见性，不自动包含子部门。
- 支持登录用户从侧边栏 footer 进入隐藏页面查看自己可见的已发布通知公告。
- 展示页按类型使用两个 tab：通知、公告。
- 每个 tab 内支持关键词搜索。
- 点击列表项打开详情抽屉展示 HTML 正文。
- 服务端从 Tiptap JSON 派生并保存 `contentHtml`，展示接口直接返回 HTML。
- 对 HTML 输出做 sanitizer，确保展示边界安全。

## 非目标

本轮不实现：

- 独立后台首页或工作台。
- 无需登录的公开公告页面。
- 已读/未读状态、未读角标、阅读确认、回执或统计。
- 公告评论、点赞、收藏、分享。
- 定时发布、发布审批、变更历史。
- 部门父子级联可见、排除规则或复杂条件表达式。
- 附件、图片上传或新的富文本能力。

## 方案选择

本轮采用“可见范围模型 + 我的公告隐藏页”的方案。

备选方案中，“只做全员公告展示”实现更快，但无法满足“只有选定的人才能查看”的需求；“公告中心 + 已读确认 + 统计”功能完整但超出本轮展示目标。当前方案能一次性补齐数据边界、展示接口和用户入口，同时避免引入过重的消息中心能力。

## 可见范围

公告新增显式可见范围：

- `all`：全员可见。
- `targeted`：指定可见对象。

指定可见对象支持三类：

- 用户：当前登录用户 ID 命中即可见。
- 部门：当前登录用户直属部门 ID 命中即可见。
- 角色：当前登录用户角色 ID 命中即可见。

多类可见对象之间使用 OR 规则：命中任一用户、部门或角色即可查看。

草稿允许暂不配置可见范围，方便编辑过程中保存。发布时必须满足以下条件之一：

- 可见范围为全员可见。
- 可见范围为指定可见对象，并且至少选择一个用户、部门或角色。

如果发布校验失败，服务端返回字段级错误，前端在管理表单中展示明确提示。

## 部门树选择规则

部门选择使用树形控件，便于发布人理解组织结构和搜索定位。树形控件不启用父子级联：

- 勾选父部门时，只表示父部门本身被选中。
- 勾选子部门时，不自动勾选父部门。
- 父部门不自动包含子部门用户。
- 半选状态如果由 UI 控件展示，只作为视觉汇总，不写入后端，也不参与可见性判断。

该规则让发布边界保持清楚：发布人选择了哪些部门，就只有这些部门的直属用户可见。

## 数据模型

`content_announcements` 新增字段：

| 字段 | 说明 |
| --- | --- |
| `visibility` | 可见范围，取值为 `all` 或 `targeted`。 |
| `content_html` | 从 `content_json` 派生并经过 sanitizer 的 HTML 正文。 |

新增关系表 `content_announcement_targets`：

| 字段 | 说明 |
| --- | --- |
| `announcement_id` | 关联 `content_announcements.id`。 |
| `target_type` | 可见对象类型：`user`、`department`、`role`。 |
| `target_id` | 对应用户、部门或角色 ID。 |
| `created_at` | 创建时间。 |

建议约束和索引：

- 主键或唯一索引：`announcement_id + target_type + target_id`。
- 索引：`announcement_id`。
- 索引：`target_type + target_id`。

`contentJson` 仍是正文事实来源，前端不提交 HTML。服务端在创建公告和更新正文时同步派生：

- `contentText`：用于搜索和空正文校验。
- `contentHtml`：用于展示页详情。

如果编辑公告时没有修改 `contentJson`，则保留原 `contentText` 和 `contentHtml`。迁移需要为已有公告补齐 `content_html`，避免旧数据展示为空。

## 契约设计

公告管理契约增加：

- `visibility`。
- `targets`，数组项包含 `targetType` 和 `targetId`。
- 详情响应返回可见范围和可见对象摘要，用于表单回填。
- 管理详情可以返回 `contentHtml`，但编辑器仍只使用 `contentJson`。

新增展示契约：

- 展示列表查询：`page`、`pageSize`、`type`、`keyword`。
- 展示列表项：`id`、`type`、`title`、`summary`、`pinned`、`publishedAt`。
- 展示详情：列表项字段加 `contentHtml`。

展示契约不返回 `contentJson` 和 `contentText`，避免把编辑态数据暴露给展示页。

## API 设计

管理接口继续挂在 `/api/content/announcements`，并继续使用现有管理权限：

- `content:announcement:list`
- `content:announcement:create`
- `content:announcement:update`
- `content:announcement:delete`

管理接口行为调整：

- 创建、编辑时接收可见范围和可见对象。
- 发布时校验可见范围。
- 管理列表仍返回全部公告给有管理权限的用户。

新增登录用户展示接口：

- `GET /api/content/my-announcements`
- `GET /api/content/my-announcements/:id`

展示接口只要求登录，不要求 `content:announcement:*` 管理权限。展示接口只返回：

- 已发布公告。
- 当前登录用户可见的公告。
- 未软删除公告。

展示详情中，如果公告不存在、未发布、已删除或当前用户不可见，统一返回 404，避免暴露公告存在性。

## HTML 派生与安全

服务端基于当前启用的 Tiptap 扩展从 `contentJson` 派生 HTML，并在写入 `content_html` 前进行 sanitizer。

原则：

- 不信任客户端提交的 HTML。
- 不允许前端直接提交 `contentHtml`。
- `contentHtml` 只能由服务端从已校验的 Tiptap JSON 生成。
- 展示页使用服务端返回的 `contentHtml` 渲染正文。
- sanitizer 规则应覆盖链接等富文本输出，避免危险属性或脚本进入展示页面。

## 前端管理端设计

`AnnouncementFormDrawer` 增加“可见范围”区域：

- 单选项：`全员可见`、`指定可见对象`。
- 选择 `指定可见对象` 后展示三个控件：
  - 用户多选。
  - 部门树形多选，精确选择，不级联。
  - 角色多选。

草稿保存允许可见对象为空。保存并发布、列表发布时需要服务端校验可见范围，并把错误反馈到表单或消息提示中。

管理列表可以增加可见范围摘要：

- `全员可见`
- `指定可见对象`

完整可见对象只在编辑抽屉中回填和维护，不在列表中展开展示。

## 前端展示页设计

侧边栏 footer 增加“通知公告”入口，和“个人设置”同级：

- 展开侧边栏时，在用户信息右侧展示图标按钮。
- 收起侧边栏时，展示独立圆形图标按钮。
- tooltip 和页面标题使用“通知公告”。

新增隐藏路由：

- `/account/announcements`

该路由不进入权限菜单资源，但需要登录。它与个人设置一样属于登录用户入口。

页面结构：

- 标题：通知公告。
- 顶部 tab：通知、公告。
- 每个 tab 内保留一个关键词搜索框。
- 列表展示当前用户可见且已发布的公告。
- 列表排序：置顶优先，发布时间倒序。
- 列表项展示：置顶标识、标题、摘要、发布时间。
- 点击列表项打开右侧详情抽屉。
- 详情抽屉展示：标题、类型、发布时间、摘要、HTML 正文。

页面不展示未读状态、不展示角标，也不承担工作台职责。

## 错误处理

管理端：

- 发布时没有合法可见范围，返回字段级错误。
- 指定的用户、部门或角色不存在、已删除或禁用，保存或发布拒绝。
- HTML 派生或 sanitizer 失败时，返回正文相关错误。

展示端：

- 列表加载失败显示错误提示。
- 详情加载失败显示错误提示或关闭抽屉。
- 不可见详情统一按 404 处理。
